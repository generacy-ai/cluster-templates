# Research: Eliminate duplicate repo clone in devcontainer mode

## Technology Decisions

### 1. Devcontainer Detection Method

**Chosen**: Check `DEVCONTAINER` and `REMOTE_CONTAINERS` environment variables.

**Rationale**: VS Code Dev Containers extension sets these env vars automatically inside the container. This is the standard, documented approach for detecting devcontainer mode without adding custom markers.

**Alternatives considered**:
- Check for `/.dockerenv` — too generic, present in all Docker containers
- Check for `/workspaces` mount — also present in standalone compose mode
- Custom env var in devcontainer.json — requires user configuration

### 2. Clone Path Derivation

**Chosen**: `REPO_NAME=$(basename "${REPO_URL%.git}")` → `/workspaces/${REPO_NAME}`

**Rationale**: Matches the path convention used by VS Code Dev Containers extension, which clones to `/workspaces/<repo-name>`. Using `%.git` parameter expansion strips the `.git` suffix if present, and `basename` extracts just the repo name from the full URL.

**Edge cases**:
- URL without `.git` suffix: `basename` still works correctly
- URL with trailing slash: `basename` handles this
- SSH URLs (`git@github.com:org/repo.git`): `basename` extracts `repo.git`, then `%.git` strips suffix → `repo`

### 3. Shared Helper vs Inline Fix

**Chosen**: Extract `resolve-workspace.sh` as a shared sourced script.

**Rationale**: The clone/detect logic is identical across 4 entrypoint files (2 variants × 2 roles). A shared helper:
- Ensures consistency — fix once, apply everywhere
- Reduces merge conflict surface
- Makes the entrypoint scripts cleaner and focused on their role-specific logic

**Alternatives considered**:
- Inline the fix in each file — simpler but requires maintaining 4 copies
- Base entrypoint script with role-specific overrides — over-engineered for this scope

### 4. Config Path Passing Strategy

**Chosen**: Conditional `--config` flag based on file existence check.

**Rationale**: The `generacy setup workspace` command already supports `--config`. By checking `[ -f "$CONFIG_PATH" ]` before passing the flag, we maintain backwards compatibility with workspaces that don't have a `.generacy/config.yaml`.

## Implementation Patterns

### resolve-workspace.sh Pattern

```bash
#!/bin/bash
# Resolve workspace directory: devcontainer detection + clone logic
# Sources into calling script — exports WORKSPACE_DIR

resolve_workspace() {
    local repo_url="${REPO_URL:-}"
    local repo_branch="${REPO_BRANCH:-main}"

    # 1. If WORKSPACE_DIR is explicitly set, use it
    if [ -n "${WORKSPACE_DIR:-}" ]; then
        log "Using explicit WORKSPACE_DIR: ${WORKSPACE_DIR}"
        return 0
    fi

    # 2. Derive repo name from URL
    if [ -n "$repo_url" ]; then
        local repo_name
        repo_name=$(basename "${repo_url%.git}")
        export WORKSPACE_DIR="/workspaces/${repo_name}"
    else
        export WORKSPACE_DIR="/workspaces/project"
        return 0
    fi

    # 3. Devcontainer detection — skip clone if repo exists
    if [ -n "${DEVCONTAINER:-}" ] || [ -n "${REMOTE_CONTAINERS:-}" ]; then
        if [ -d "${WORKSPACE_DIR}/.git" ]; then
            log "Devcontainer mode: using existing repo at ${WORKSPACE_DIR}"
            return 0
        fi
    fi

    # 4. Clone or pull
    if [ ! -d "${WORKSPACE_DIR}/.git" ]; then
        log "Cloning project repo: ${repo_url} (branch: ${repo_branch})"
        git clone --branch "$repo_branch" "$repo_url" "$WORKSPACE_DIR"
    else
        log "Project repo already cloned, pulling latest..."
        cd "$WORKSPACE_DIR"
        git fetch origin
        git pull --ff-only origin "$repo_branch" 2>/dev/null || true
    fi
}

resolve_workspace
```

### Config-Aware Setup Pattern

```bash
CONFIG_PATH="${WORKSPACE_DIR}/.generacy/config.yaml"
if [ -f "$CONFIG_PATH" ]; then
    generacy setup workspace --config "$CONFIG_PATH" --clean 2>>"$SETUP_LOG" || ...
else
    generacy setup workspace --clean 2>>"$SETUP_LOG" || ...
fi
```
